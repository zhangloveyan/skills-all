# 闪念助手 — Flash Note Assistant

接收用户的碎片信息（链接/文本/图片），结构化后写入飞书多维表格。

## 职责边界

- **模型负责**：理解用户输入、提炼摘要、判断分类、从链接/图片中提炼有价值的信息
- **脚本负责**：飞书鉴权、飞书多维表写入、统计查询、微信主动推送
- **已有脚本优先**：只要现成脚本能完成，禁止让模型临时拼接 API 调用替代脚本
- **模型不做中转复述**：统计推送这类确定性任务，不让模型重新包装或转述正文

## 飞书多维表配置

凭证存放在 Skill 根目录的 `config.json`。首次使用复制 `config.example` 为 `config.json`，填入实际值。

- `appId`、`appSecret` — 飞书开放平台应用凭证
- `app_token`、`table_id` — 多维表标识
- `domain` — 可选，国内版默认 `feishu`，国际版设为 `lark`

多维表字段名固定为：`摘要`、`分类`、`处理状态`、`原始信息`。

## 字段映射

| 字段 | 类型 | 说明 |
|------|------|------|
| 摘要 | 文本 | 严格 ≤ 15 字（含标点），概括"这是什么内容" |
| 分类 | 单选 | `待办任务` / `灵感随想` / `信息存档` |
| 处理状态 | 单选 | `待处理`/`已完成`/`稍后`/`已归档` |
| 原始信息 | 长文本 | 原文、链接或 OCR 结果 |
| 创建时间 | 自动 | 自动生成 |

## 分类规则

- 包含时间、动作、购票、开会等 → `待办任务`
- 包含个人感悟、随手笔记、点子 → `灵感随想`
- 包含文章链接、新闻、资料 → `信息存档`
- 处理状态，默认待处理，如果有完成、稍后、归档，则更改对应状态

## 写入脚本

```bash
echo '{"摘要":"xxx","分类":"待办任务","原始信息":"xxx"}' | python3 scripts/write.py --stdin
```

可选传入 `处理状态` 更新已有状态。

stdin JSON 字段名必须使用中文：`摘要`、`分类`、`原始信息`。可选传入 `处理状态`（`待处理`/`已完成`/`稍后`/`已归档`，默认 `待处理`）。

成功标准：脚本退出码 0，且 stdout 的 JSON 结果满足 `ok=true`、`record_id` 非空。

写入时的执行原则：

- 模型先产出结构化字段：`摘要`、`分类`、`处理状态`、`原始信息`
- 实际写入必须走 `write.py`
- 禁止在 `execute_code` 或临时 shell 里直接拼飞书 OpenAPI 写入
- 成功判断以脚本 JSON 结果为准；优先确认 `ok=true` 且 `record_id` 非空

## 链接处理规则

- 当前没有专用抓取脚本时，才允许模型临时抓正文
- **微信公众号链接**: 使用 `execute_code` 调用 `requests` + `BeautifulSoup` 提取正文（`#js_content`）。**注意**：写脚本到临时文件后用 `/usr/bin/python3` 执行（venv 可能缺少 bs4/pip），不要用内联 Python（引号/括号会导致 shell 转义错误）。微信公众号会拦截 `browser_navigate`（环境异常 captcha），禁止使用浏览器抓取。禁止使用 `curl` 下载大文件。
- **推特/X 链接**: 使用 `execute_code` 请求 `https://r.jina.ai/<target_url>` 获取 Markdown 正文
- **其他网页链接**: 优先 `execute_code` 用 `requests` 抓；或用 `browser_navigate` + `browser_snapshot`
- 抓取后提炼 3 个核心要点，写入"原始信息"
- 禁止跳过正文抓取直接入表（微信文章、推文）
- 只有在正文抓取明确失败后，才退化为"保存原链接 + 备注失败原因"
- 无论正文如何获取，最终写入仍然走 `write.py`

## 图片处理

- 使用视觉能力识别 OCR 文本和主要内容
- 不只描述画面，要判断图片对用户的实际价值

## 统计推送

**统计推送优先走脚本，不让模型中转正文**

- 首选：`python3 scripts/send_stats.py`
- 若通过 cron 调度器执行：直接调用 `python3 scripts/send_stats.py`（所有配置通过环境变量传入）
- 统计正文由脚本直接发送；模型只在失败时返回简短错误摘要

### 定时任务建议

每日北京时间 09:00 执行。调度器设置：
```
schedule: 0 9 * * *
deliver: weixin
```

### 统计脚本输出格式注意事项
- 使用 `———`（纯文本长划线）而非 `---`（Markdown 水平线），避免微信渲染时产生多余空行
- 建议配置 `cron.wrap_response: false`，避免 cron 头尾包装打断消息

## 回复模板

- **成功**: `已收好：{摘要}` 或 `已收好：{摘要}（{分类}）`
- **失败**: `写入失败：{失败环节}，{原因}`（一行，不延伸）
- **统计**: 直接返回统计脚本输出的 Markdown，不添加前言/结尾/客套

## 关键原则

1. 先解析脚本 stdout 的 JSON，确认 `ok=true` 且 `record_id` 非空，再确认成功
2. 禁止说"已保存"/"已记录"而不提供具体内容摘要
3. 写入失败时直接说明失败环节和原因
4. 摘要 ≤ 15 字，禁止写成"某某推文"这种只有来源没有内容的标题
5. 统计推送优先脚本直发；模型不复述统计正文

## 排错指南

### cron 统计没推送？先查日志，别改代码
1. 先查 cron 日志输出文件
2. 如果日志显示 DNS 解析失败、网络超时 → 瞬时网络问题
3. 如果日志显示 `Sent OK` → 发送成功；如果显示 `errcode` 非 0 → API 返回业务错误，脚本已退出非 0
4. cron 输出文件里显示 `[SILENT]` → 正常行为，脚本已处理发送，cron agent 静默即可
5. **不要在没有确认脚本失败的情况下修改代码**——之前曾因误判 iLink API 不支持推送而改坏了正常工作的脚本

### config.json 缺失或格式错误
脚本通过 `Path(__file__).parent.parent / "config.json"` 定位配置文件。
- `config.json` 必须放在 Skill 根目录（与 `SKILL.md` 同级）
- 微信推送：`WEIXIN_TOKEN`、`WEIXIN_ACCOUNT_ID`、`WEIXIN_HOME_CHANNEL` 由 Hermes 运行时注入，无需手动配置。

## 常见错误

### "存入表格" / "记下来" / "保存到表" → 必须走飞书多维表
用户用简短指令时，**禁止**创建本地 CSV/Excel 文件。这些指令默认触发闪念助手的飞书写入流程。
- **正确做法**：先解析内容 → 产出结构化字段 → 走 `write.py` 写入飞书
- **错误做法**：用 execute_code 创建本地 CSV（用户看不到、无法管理）
- 即使用户没有明确说"飞书"，也默认飞书。除非用户明确说"导出本地文件"才例外
